Memory as reliability practice

I built this after noticing the same context tax over and over: a good AI session solved the immediate problem, but the useful residue did not make it into my operating system.

Role lensEngineering manager (SRE background)
Artifactmemory plugin
ScopeWork and personal systems

When a hard problem finally gets solved, the fix is rarely enough. Someone has to preserve the lesson in a way the next person can actually use.

That same pattern shows up in AI-assisted work. A session can be productive and still leave almost no durable residue. The assistant helped debug a strange failure, compare trade-offs, draft a review, or clarify a decision. Then the session ends. A week later, I remember that we figured it out, but not the exact shape of the answer.

That is fine for a throwaway task. It is expensive for real work. It is especially expensive for managers, because a lot of the job is continuity: what changed, why it changed, who needs context, what risk is still open, and which lesson should survive the week.

This is not a note-taking project. It is a small project for keeping my own context reliable.

Why I built it

I wanted a memory system that behaves like infrastructure I would trust: local-first, inspectable, reversible, and explicit about writes. I did not want an opaque memory feature that quietly shaped future sessions. I wanted plain Markdown, git history, and a setup flow that asks before touching dotfiles.

The result is a public plugin called memory. Its main skill, /memory:memory-setup, scaffolds a git-backed Obsidian vault, installs the useful Obsidian pieces, and wires session capture into Claude Code. The repo also includes compatibility notes and importers for Codex and Antigravity, because the useful context should belong to the person doing the work, not to one assistant vendor.

Two optional layers round it out. Scheduled sync keeps the vault converged across machines — obsidian-git handles it while Obsidian is open, and a launchd agent pulls, commits, and pushes headlessly the rest of the time. And invoking the skill with the crypt argument encrypts the sensitive directories (coaching notes, session logs, meeting notes) with git-crypt, so the remote only ever sees ciphertext while everything stays plain Markdown on disk; the key lives in your password manager, and a second machine unlocks with one command.

I have used this pattern in both work environments and personal environments. At work, it helps with project follow-ups, architecture reviews, team handoffs, vendor decisions, team preferences, and the small operational facts that otherwise get rediscovered every month. Personally, it helps with coaching notes, side projects, routines, reading trails, and decisions I keep revisiting.

Work memory

  • Debugging context and follow-ups after hard problems
  • Handoffs and open-risk notes
  • Architecture decisions and review patterns
  • Team conventions that should not live only in chat

Personal memory

  • Coaching reflections and recurring themes
  • Personal projects and unfinished threads
  • Learning notes from AI, systems, and leadership
  • Habits, routines, and decisions worth revisiting

The part worth copying

The exact folder names matter less than the operating pattern. Start each assistant from a small index. Keep raw session exhaust out of the default context path. Promote only the facts, decisions, references, and preferences that are likely to help later.

That gives the system a useful bias: capture can be broad, but memory should be curated. The skill exists to make that setup easy to inspect, easy to repeat, and hard to confuse with a magic black box.

Why continuity is worth engineering

Continuity is not just about uptime. It is the discipline of making work understandable when the easy context is gone. Good notes, docs, decision records, and handoffs all do the same thing: they reduce the time it takes the next person to get oriented and act.

AI work needs some of that discipline. Agents are fast, but speed can hide context loss. If the assistant helps me reason through a tricky production question and the output never becomes durable, the organization learned less than it could have. If a coaching session surfaces a repeated pattern and I never capture it, I am asking future-me to do the same discovery again.

So I built the skill with the same guardrails I would want on any system I trust. It explains the trade-off first. It runs read-only preflight checks. It asks for the vault name, location, and remote. It backs up settings before editing. It makes the managed block visible. It keeps the recovery path boring.

Influences: visible machinery

I have been drawn to Andrej Karpathy's teaching style because it favors small systems you can inspect. Neural Networks: Zero to Hero teaches neural networks from scratch in code. micrograd makes backpropagation small enough to hold in your head. nanoGPT makes the training loop legible without burying the reader in platform machinery. I like that posture: build enough of the thing end to end that you can reason about it without mythology.

This memory skill is not trying to build an LLM. It borrows that operating principle: make the important machinery visible enough that you can debug it, trust it, and change it.

The practical test: if this is working, a future session should need less preamble, fewer repeated explanations, and fewer “wait, where did we decide that?” moments.

Try it, inspect it, adapt it

I published the memory plugin as a small, readable starting point for anyone who wants local AI-session memory without adopting a whole knowledge-management framework.

The important pieces are plain files: Markdown templates, Python scripts, and explicit setup steps. Read the README, inspect the writes, and change the workflow until it fits your own environment.

Prerequisites

Before running the setup skill, confirm these tools are available on your machine. The skill can install some of them automatically, but having them ready avoids interruptions.

Claude Code CLI

Requires Node.js ≥ 18.

npm install -g @anthropic-ai/claude-code

Or download from claude.ai/code.

Homebrew (macOS)

Needed if any tool installs fall back to Homebrew.

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Already installed but hitting "shallow clone" errors? Run: git -C $(brew --repository)/Library/Taps/homebrew/homebrew-core fetch --unshallow and git -C $(brew --repository)/Library/Taps/homebrew/homebrew-cask fetch --unshallow

GitHub CLI (gh)

Needed to create the private vault repo. The skill installs it automatically if missing, but to install manually:

brew install gh
gh auth login

Or download a release directly from github.com/cli/cli/releases (macOS: .zip for amd64 or arm64).

Auto mode

Enable Auto mode in Claude Code before running setup to avoid approval prompts during installation. Type in Claude Code:

/auto

Disable it after setup completes by typing /auto again.

Install the plugin

Add the public marketplace, install the memory plugin, then run the setup skill.

Codex

codex plugin marketplace add adzuci/claude-plugins
codex plugin add memory@adzuci-plugins
/memory:memory-setup

Claude Code

# 1. Install Claude Code (requires node ≥ 18)
npm install -g @anthropic-ai/claude-code

# 2. Add the plugin marketplace
claude plugin marketplace add adzuci/claude-plugins

# 3. Install the memory plugin
claude plugin install memory@adzuci-plugins

# 4. Enable Auto mode (recommended — avoids approval prompts)
# Type in Claude Code: /auto

# 5. Run setup
# Type in Claude Code: /memory:memory-setup

During setup: When the skill asks you to run gh auth login, type ! gh auth login directly into the Claude Code prompt — the ! prefix runs it in your terminal so the browser flow works correctly.

Tested on macOS. The skill installs Obsidian, the GitHub CLI, and jq automatically if missing. Linux and Windows compatibility PRs are welcome.

More plugins

The marketplace includes two additional plugins beyond memory.

claude plugin install productivity@adzuci-plugins

Job market intelligence sweep for senior technical candidates. Produces a tier-ranked report in your vault, runs an automated recruiter review, and offers weekly scheduling.

claude plugin install career@adzuci-plugins

What I want from it

I want future sessions to start with better orientation. I want decisions to have a place to land. I want patterns from work and life to become easier to see across time. And I want the memory layer to remain mine: plain files, local defaults, clear writes, and enough structure to be useful without becoming ceremony.

That is the practical reason. The leadership reason is simpler: if AI is becoming part of how I think, build, and manage, I do not want the context layer to be accidental.
